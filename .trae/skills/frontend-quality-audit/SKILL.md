# 前端质量审计

这个 skill 用于审计前端项目中的工程质量问题，但默认应该保持“够用、直接、方便人工修复”，不要把报告做成复杂的治理平台。

以下场景优先调用：

- 团队准备在重构前做一次质量体检
- UI 工程师参与前端开发，但模块拆分和边界控制较弱
- 项目中出现 1000 到 2000 行的大文件，或一个文件同时承担页面、状态、接口和工具职责
- 团队希望输出正式的 Markdown 或 HTML 报告，而不是零散结论
- 用户要求对 React、Vue、Angular、TypeScript 或 JavaScript 项目做治理型分析

## 默认目标

默认只做最重要的 4 件事：

1. 找出超过 800 行的业务文件，`.vue` 优先
2. 找出疑似未删除的大段代码，并标出行号范围
3. 找出重复函数和相似逻辑，整理成可执行的小案例
4. 生成一份开发者能直接照着修的 `.md` 或 `.html` 报告

只有在用户明确要求时，才增加评分卡、驾驶舱、风险地图、治理路线图这类增强内容。

## 输出位置

报告默认必须写到“被审计项目目录”里，而不是 skill 自己所在的 `skills` 目录。

- `--projectPath` 必须指向真正要审计的项目根目录
- 如果工具运行在 skill 目录中但没有传 `--projectPath`，应直接报错，不要把报告写到 skill 目录
- 如有需要，可通过 `--outputDir` 显式指定报告输出目录，但这个目录也应位于被审计项目内

## 审计范围

默认优先检查以下维度：

- 超过 800 行的业务文件，优先看 `.vue`、页面组件、业务组件
- 无用代码或可疑残留
- 重复方法、相似函数块
- 职责混杂
- 只在必要时补充工具链和评分信息

## 工作方式

请按下面这个最小流程推进：

1. 先找“大文件”
   - 列出所有超过 800 行的业务文件
   - `.vue`、页面组件、业务组件优先显示
   - 最多输出前 30 个文件
2. 再找“没清理掉的东西”
   - 重点标出疑似整段注释旧代码
   - 必须标出文件、起止行号
   - 最多输出前 30 个文件
3. 再找“值得人工拆的地方”
   - 重复函数
   - 相似函数块
   - 职责混杂文件
   - 每个命中点都要整理成一个小案例
4. 最后生成交付物
   - 生成 `前端质量审计报告.md` 或 `前端质量审计报告.html`
   - 默认报告应该短、直接、按修复顺序排

## 报告必备结构

每份报告默认只需要包含：

1. 一句话摘要
2. 第一步：超过 800 行的业务文件
3. 第二步：疑似未删除的大段代码
4. 第三步：重复函数 / 相似逻辑案例
5. 第四步：报告产物
6. 可选补充信息

评分卡和等级不是必需项，只有在用户明确需要“治理视角”时再加。

## 报告规则

- 优先给出证据，而不是抽象判断
- 必须包含准确文件路径
- 能给行号时尽量给行号
- 优先使用开发者看得懂的话
- 不要把报告做成展示面板，重点是方便照着修

## 建议扫描文件模式

优先：

- `src/**/*.ts`
- `src/**/*.tsx`
- `src/**/*.js`
- `src/**/*.jsx`
- `src/**/*.vue`
- `src/**/*.scss`
- `src/**/*.css`

弱化或忽略：

- `node_modules`
- 构建产物
- 覆盖率产物
- 生成代码
- 锁文件

## 可选自动化

默认只保留 TypeScript 工具这一条主路径：

```powershell
cd ".trae/skills/frontend-quality-audit/tool"
npm install
npm run audit -- --projectPath "<PROJECT_PATH>" --outputFormat both
```

如果希望把报告放到项目内的固定子目录，可以显式传入：

```powershell
npm run audit -- --projectPath "<PROJECT_PATH>" --outputFormat both --outputDir "<PROJECT_PATH>\\.audit-reports"
```

TypeScript 工具更适合后续接入 ESLint、TypeScript、AST、依赖图和更复杂的 HTML 渲染逻辑。

## 手工模式

如果不想依赖工具，也可以直接使用这个 skill 进行手工审计并生成报告：

1. 先通过代码检索定位热点文件、规则绕过、调试残留、超大文件和职责混杂点
2. 再按优先级整理为“立即处理项”“重构候选清单”“热点文件”“拆分建议”
3. 最后直接填充 `resources/report-template.md` 或 `resources/report-template.html` 的结构，生成最终交付物

这种方式更适合：

- 项目结构非常特殊，难以依赖统一规则
- 团队更重视人工判断和架构经验
- 希望先快速产出高质量 HTML/Markdown 报告，再决定是否接入自动化

## Markdown 模板

需要手工整理报告时，可使用 `resources/report-template.md` 作为结构参考。

## HTML 模板

需要可展示的样式化报告时，可使用 `resources/report-template.html` 作为结构参考。

## 最终交付标准

一份合格的最终交付至少应包含：

- 一眼能看懂先修什么
- 能直接定位到文件，最好还能定位到行号
- 不需要读完整个报告，也能开始动手修
- 保存到目标项目或当前工作区中的实际报告文件
